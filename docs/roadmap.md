# Roadmap

这个文档记录远程渲染桌面小屏的后续方向。这里保持轻量，只放产品和架构层面的路线；具体协议、性能和实现细节放在其他文档里。

## 近期

- 把当前中文桌面时钟打磨成稳定日常界面：更清晰的状态提示、更稳的错误降级、更少干扰的同步反馈。
- 继续完善 Settings 的真实控制项和只读诊断项，优先考虑能直接帮助部署、排障和日常使用的能力。
- 强化配置链路：AP fallback、局域网配网页、远端服务地址修改、设备 ID 修改和远端不可用时的本地提示都要保持可靠。
- 保持固件职责足够小：WiFi、配网页、按键、状态上报、命令执行和 TFT 输出。
- 继续用本地 PNG 预览工具验证 UI 改动，再上设备做显示效果和时序确认。

## 远端渲染服务

- 增加可配置的首页小组件，例如短句、提醒、农历、节气等。
- 天气保持可选，不作为核心功能；如果加入，也由服务端渲染，并让失败状态尽量安静。
- 抽象更好用的 Python 布局辅助能力，减少页面开发时大量手写坐标。
- 动画继续优先控制 payload，大面积变化尽量拆成较小 dirty region。
- 为常开部署补齐更清楚的运行检查，例如服务健康、设备状态新鲜度、最近帧信息和预览输出。

## 设备固件

- 继续把 HTTP Keep-Alive 帧拉取作为稳定传输路径，除非新协议能证明有明确收益。
- 避免 ESP8266 上的全屏帧缓冲，继续用流式读取和小批量 TFT 输出控制 RAM 压力。
- 保留并扩展客户端状态上报，把有用的设备侧诊断指标同步给远端 UI。
- 优先增加小而明确的命令类型，例如重启、旋转、诊断采集或配置刷新。
- 保持 `src/AppConfig.h` 为编译期默认值入口，运行时可变配置继续通过设备配网页持久化。

## 部署

- 补充本地 Docker 之外的部署方式，例如局域网内常开的轻量服务器、NAS 或小主机。
- 远端渲染服务始终通过 URL 配置，不假设特定主机、固定 IP 或具体网络环境。
- 为长期运行补齐基础可观测性：健康检查、日志轮转、设备最近在线时间和简明故障提示。

## 可能的后续想法

- 首页自定义短句或小提醒。
- 多种时钟样式或主题。
- 服务端页面/组件的轻量插件机制。
- 更多设备命令，例如旋转、重启、诊断采集等。
