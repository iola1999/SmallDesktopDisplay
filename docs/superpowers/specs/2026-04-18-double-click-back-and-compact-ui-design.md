# SmallDesktopDisplay 双击返回与紧凑导航 UI 设计

## 背景

上一轮已经把单按钮设置框架、后台静默刷新、首页动画作用域、长按反馈这些基础能力接上，当前固件已经具备：

- 首页短按提示气泡
- 长按进入设置
- 设置页短按切换、长按执行
- 诊断页短按滚动、长按返回
- 亮度页短按切换、长按保存
- 长按顶部进度线反馈

但真机体验上还暴露出一组新的交互与 UI 问题：

1. 用户希望利用“双击”作为新的返回手势，减少页面上持续占位的返回提示。
2. 当前顶部标题区整体偏高，小字说明过长，导致信息密度不合理。
3. 当前底部操作引导代码虽然存在，但真机上基本不可见，等价于失效。
4. 当前长按阈值过短，容易误触发，缺少“确认动作应该更稳一点”的手感。

这轮工作的目标不是再加一个孤立手势，而是把“短按前进、长按确认、双击返回”收敛成新的统一输入语义，同时把顶部/底部导航 chrome 压缩成更轻、更稳、更可读的一套样式。

## 目标

1. 为所有非首页页面引入双击返回能力。
2. 保持单击为“下一项 / 滚动 / 切换”，长按为“确认 / 执行”，不改变首页双击无动作这一规则。
3. 将长按阈值从当前约 `200ms` 提高到 `500ms`。
4. 缩小顶部标题区整体尺寸，只保留非常小的标题 UI，不再显示长句副标题。
5. 重做底部操作引导，使其以很小高度稳定显示在几乎所有非首页页面。
6. 保持现有页面结构和功能边界，不借这轮去重做设置系统或视觉风格。

## 非目标

- 这轮不新增新的设置菜单项。
- 这轮不改后台刷新逻辑本身。
- 这轮不重做首页主体布局。
- 这轮不引入多按钮方案。
- 这轮不支持首页双击动作。
- 这轮不把底部引导做成复杂图标或动画组件。

## 约束与已知事实

### 1. 双击可以识别，但会引入短按确认延迟

当前使用的 `Button2` 已经支持：

- `setClickHandler`
- `setDoubleClickHandler`
- `setLongClickHandler`

一旦接入双击，单击回调会等待双击判定窗口结束后才真正触发。  
这意味着所有“短按下一项 / 短按滚动 / 短按切换”的动作都会比现在多一个约 `300ms` 的确认延迟。

用户已经明确接受这个取舍，因此这轮设计不再回避这一点，而是围绕这个前提统一输入模型。

### 2. 底部引导当前不是“没有代码”，而是“基本不可见”

当前底部引导已经有统一绘制入口，但位置太靠下、对比度和存在感也偏弱，因此在真机上很容易被理解为完全没有显示。

所以这轮不是“新增一个底部框架”，而是修复已有框架的可见性与信息密度。

## 方案比较

### 方案 A：只新增双击返回

仅把双击回退接入输入层和页面状态机，其余顶部/底部 chrome 基本不动。

- 优点：改动小。
- 缺点：顶部拥挤和底部不可见的问题仍然存在，交互收束不完整。

### 方案 B：双击返回 + 紧凑导航 chrome 一起收口

把输入语义、长按阈值、顶部标题区、底部操作引导一次性统一。

- 优点：行为和视觉都一起收口，后续页面新增不会再重复修补。
- 缺点：因为启用双击，所有短按动作会出现轻微确认延迟。

### 方案 C：顶部极简化，底部提示全部去掉

只保留标题，不再做任何页面操作引导。

- 优点：最干净。
- 缺点：学习成本高，不适合当前仍在快速迭代交互模型的阶段。

### 结论

采用方案 B。  
也就是：

- 输入语义统一为“短按前进、长按确认、双击返回”
- 长按阈值调到 `500ms`
- 顶部 chrome 整体缩小
- 底部改为极简、稳定可见的统一操作引导

## 新交互模型

### 首页

- 短按：显示调试气泡
- 长按：进入设置
- 双击：无动作

首页保持现有逻辑，不引入返回语义。

### 菜单页

- 短按：选中下一项
- 长按：确认当前项
- 双击：返回上一层

适用于：

- 设置菜单
- 重启确认页
- 后续新增的其他菜单型页面

### 内容页

- 短按：向下滚动 / 焦点下移
- 长按：确认当前动作；如果页面没有独立确认动作，则等价于返回
- 双击：返回上一层

当前诊断页没有独立确认动作，因此长按与双击都回退。  
这不冲突，因为长按仍然保持“确认”的统一语义，只是在该页的确认动作恰好就是返回。

### 调节页

- 短按：切换下一档 / 下一项
- 长按：确认保存
- 双击：返回上一层且不保存

当前亮度页即属于这一类。

## 顶部导航 UI 设计

### 设计目标

- 减少顶部 chrome 垂直占用
- 不再依赖长句副标题描述操作
- 让标题只是“页面名”，而不是一块重 UI

### 具体规则

1. 顶部标题容器整体缩小高度。
2. 标题字号缩小一级。
3. 默认不再显示长句副标题。
4. 顶部只保留非常小的标题 UI，不承担完整操作说明职责。
5. 长按进度线继续存在，但要与缩小后的标题区兼容。

### 文案策略

顶部不再显示：

- `Long press to enter`
- `Hold to execute`
- `Tap to scroll`
- `Preview before save`

这类完整句子全部移除。  
顶部只显示页面名称，例如：

- `Settings`
- `Restart`
- `Diagnostics`
- `Brightness`

## 底部操作引导设计

### 是否保留

保留。  
用户已经明确否定“完全删掉”，但要求高度极小、几乎不占空间。

### 设计目标

- 只占用非常薄的一条安全区域
- 作为统一引导框架复用到几乎所有非首页页面
- 文案极短、对比度清楚、位置稳定

### 显示范围

- 首页：不显示
- 菜单页：显示
- 内容页：显示
- 调节页：显示
- 确认页：显示

### 文案策略

底部引导不再写较长短语，而改成极短 token：

- 短按：`Tap`
- 长按：`Hold`
- 双击：`2x`

再配合动作词，例如：

- 菜单页：`Tap Next  Hold OK  2x Back`
- 内容页：`Tap Scroll  Hold OK  2x Back`
- 亮度页：`Tap Next  Hold Save  2x Back`
- 重启确认页：`Tap Next  Hold OK  2x Back`

要求：

1. 三段文案都必须非常短。
2. 不再使用完整说明句。
3. 位置上移到底部安全区内。
4. 对比度要显著高于当前版本。

## 长按阈值调整

### 当前问题

当前长按阈值约为 `200ms`，确认动作过于激进，容易误触发。

### 新规则

- 长按阈值调整为 `500ms`

### 影响

1. 长按确认会更稳定。
2. 顶部进度线动画时长会自然变长，更符合“确认动作”的反馈节奏。
3. 双击与长按的手感边界会更清晰。

## 数据模型

### 输入层

当前输入层已有：

- `PressStarted`
- `LongPressArmed`
- `PressReleased`
- `ShortPress`
- `LongPress`

本轮新增：

- `DoublePress`

### AppEvent

在现有事件之外新增：

- `doublePressed(nowMs)`

### ViewModel

现有 `FooterHints` 只承载短按/长按文案。  
本轮明确要求把它扩展为三段提示：

- `shortPressLabel`
- `longPressLabel`
- `doublePressLabel`

## 状态机语义

### 双击返回原则

双击不直接“撤销所有状态”，而是走与当前页面一致的回退路径：

- `SettingsMenu` -> `Home`
- `DiagnosticsPage` -> `SettingsMenu`
- `BrightnessAdjustPage` -> `SettingsMenu` 且不保存
- `RebootConfirmMenu` -> `SettingsMenu`

### 与长按的关系

长按仍然是“确认”。  
双击是显式返回。  
两者并存，不互相替代。

## 渲染层

### 顶部

- 缩小标题盒高度
- 缩小标题字号
- 默认移除副标题绘制
- 保留长按进度线与顶部盒子的层次关系

### 底部

- 上移到底部安全区
- 改成单行极简 token
- 保持统一绘制入口，不让页面各自实现

## 测试要求

### Host 侧

至少补以下测试：

1. 双击在非首页页面触发返回。
2. 首页双击无动作。
3. 亮度页双击返回时不保存。
4. 菜单页和内容页的 footer 文案包含双击返回提示。
5. 顶部副标题默认被清空或不再依赖长句。

### 嵌入式编译验证

1. `esp12e` 编译通过。
2. 长按阈值已调整为 `500ms`。
3. 双击在真机上可返回。
4. 首页双击无动作。
5. 顶部标题区明显缩小。
6. 底部操作引导在非首页页面稳定可见。

## 风险与权衡

### 风险 1：双击导致短按动作延迟

这是使用当前输入库识别双击的直接代价。  
用户已明确接受这个取舍，因此本轮不再试图规避。

### 风险 2：底部信息过多再次变挤

控制方式：

- 所有底部文案必须压缩成极短 token
- 不允许重新回到完整句

### 风险 3：长按与双击在内容页出现“两个都能返回”

这在当前内容页是可接受的，因为内容页本身没有独立确认动作。  
只要整体原则仍然是“长按 = 确认，双击 = 返回”，语义并没有失真。

## 结论

这轮工作的核心是把当前单按钮系统收束成更成熟的一套交互约定：

- 短按负责前进
- 长按负责确认
- 双击负责返回

在这套输入语义之上：

- 顶部 UI 缩小成纯标题
- 底部用极小高度承载统一引导
- 长按阈值提升到 `500ms`

这样后续无论再加设置项、更多内容页还是更多确认页，都可以落在同一套稳定的输入与 chrome 框架上。
